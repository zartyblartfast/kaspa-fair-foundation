# ENV-086 — Live TN10/Toccata workaround inventory

## Scope

This ENV adds no implementation. It identifies remaining shortcuts in the current PoC and selects the first concrete replacement target that should use real TN10/Toccata capability.

The current milestone already has Rust-owned demo artifact generation and a live read-only TN10/Toccata anchor. The remaining core gap is that the roulette round commitment and reveal are still local proof transcript data rather than live round-specific TN10/Toccata evidence.

## Category conclusions

- round commitment workaround: identified. The current commitment is generated locally and represented in JSON/proof transcript, with no live round-specific commitment transaction and no round-specific covenant transition.
- reveal workaround: identified. Reveal material is local/demo transcript data, with no live TN10 reveal transaction linked to a live commitment transition.
- explicit demo seed workaround: identified. The CLI uses `--demo-seed`; sample/proof artifacts and docs disclose explicit demo seed material. This should bind to live TN10 commitment/reveal evidence before production entropy is prioritized.
- static app-facing JSON workaround: identified. `sample-round.json` and `toccata-fairness-proof.json` are generated mirror/export artifacts copied into the UI; they should later be generated from retrieved live round evidence.
- live TN10 anchor limitation: identified. `live_readonly_tn10` evidence exists for the canonical anchor, but round-specific commitment/reveal transactions are not live.
- covenant enforcement gap: identified. Rust verifies covenant/lineage fields off-chain; the next replacement should introduce minimum viable KIP-20 lineage proof and, where feasible, KIP-17 covenant transition constraints.
- UI display limitation: identified. The UI displays generated artifacts and remains non-trusted; live evidence retrieval/generation should remain verifier-owned, not UI-owned.

## Replacement priority rationale

The highest-value next target is not production entropy and not UI fetching. It is live round-specific commitment/reveal anchoring because it removes the central proof gap: the commitment/reveal lifecycle for this roulette round is not yet TN10 evidence.

Production entropy remains important, but implementing it before live commitment/reveal anchoring would still leave the proof lifecycle local/static. UI changes should follow verified live evidence because the UI is a display surface, not the proof authority.

## Safety and authorisation

Replacing ranks 1-4 with live TN10 evidence requires explicit user authorisation before any faucet, wallet/private key, signing, transaction creation, submission, or broadcast activity. This ENV performs inventory/target selection only.
